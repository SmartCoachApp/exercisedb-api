# Internationalization (i18n)

Multi-language support for the exercises API. Allows clients to request data in different languages via a query parameter.

**Status**: Implemented

## User Capabilities

- Users can request any endpoint with `?lang=es` to get results in Spanish
- Users default to English (`en`) when no `lang` parameter is provided
- Users can search and filter using terms in either language (e.g., `?muscles=pecho` or `?muscles=chest`)
- Users can get translated: exercise names, instructions, muscle names, body part names, equipment names
- Catalog endpoints (`/bodyparts`, `/muscles`, `/equipments`) return translated values
- Fuzzy search (`/exercises/search?q=...`) works with terms in the selected language

## API Contract

All existing endpoints gain an optional `lang` query parameter:
- `GET /api/v1/exercises?lang=es`
- `GET /api/v1/exercises/search?q=pecho&lang=es`
- `GET /api/v1/exercises/filter?muscles=pecho&lang=es`
- `GET /api/v1/exercises/filter?muscles=chest&lang=es` (English filters also work with lang=es)
- `GET /api/v1/bodyparts?lang=es`
- `GET /api/v1/muscles?lang=es`
- `GET /api/v1/equipments?lang=es`
- `GET /api/v1/bodyparts/chest/exercises?lang=es` (path params always English)

Supported values: `en` (default), `es` (Spanish). Extensible to more languages.

## Bilingual Filter Support

When `lang=es`, filter parameters accept values in BOTH languages:
- `?muscles=pecho` works (Spanish term)
- `?muscles=chest` also works (English term)
- The system resolves Spanish filter values to English internally, then applies the filter, then translates the response

Resolution flow:
1. Receive filter value (e.g., "pecho")
2. Check if it's an English key -> use directly
3. If not, reverse-lookup in translation map to find English key ("pecho" -> "chest")
4. Apply filter using English key on English source data
5. Translate response to Spanish before returning

## Translation Data

### Catalog translations (88 terms)
Static dictionaries mapping English -> Spanish:
- 10 body parts (e.g., "chest" -> "pecho")
- 50 muscles (e.g., "biceps" -> "bíceps", "glutes" -> "glúteos")
- 28 equipment types (e.g., "dumbbell" -> "mancuerna")

### Exercise translations (1,500 exercises)
Per-exercise translations keyed by exerciseId:
- 1,500 exercise names (e.g., "band shrug" -> "encogimiento con banda")
- ~8,737 instruction steps

## Data Architecture

```
src/data/
├── exercises.json              # existing (English, source of truth)
├── i18n/
│   └── es/
│       ├── catalogs.json       # bodyparts, muscles, equipments translations
│       ├── exercises_001.json  # exercises 0-99 translations
│       ├── exercises_002.json  # exercises 100-199 translations
│       ├── ...
│       └── exercises_015.json  # exercises 1400-1499 translations
```

Translation files are split into 15 batches of 100 exercises each for manageable generation.
At load time, they are merged into a single in-memory map.

### catalogs.json format
```json
{
  "bodyParts": {
    "neck": "cuello",
    "chest": "pecho"
  },
  "muscles": {
    "biceps": "bíceps",
    "glutes": "glúteos"
  },
  "equipments": {
    "dumbbell": "mancuerna",
    "barbell": "barra"
  }
}
```

### exercises_NNN.json format
```json
{
  "trmte8s": {
    "name": "encogimiento con banda",
    "instructions": [
      "Paso 1: Párate con los pies separados al ancho de los hombros...",
      "Paso 2: Mantén los brazos rectos y relajados..."
    ]
  }
}
```

## Implementation Approach

1. **TranslationService** - New service that loads and caches translation files, applies translations to data
2. **FileLoader** gains `loadCatalogTranslations(lang)` and `loadExerciseTranslations(lang)` methods
3. **PaginationQuerySchema** extended with optional `lang` parameter
4. **All controllers** pass `lang` through to services
5. **Bilingual filter resolution** - Before filtering, resolve Spanish filter terms to English keys
6. **Fuzzy search on translated data** - When lang!=en, Fuse.js operates on translated exercise data
7. **Path params stay English** - `/bodyparts/chest/exercises?lang=es` returns translated data but URL uses English

## Constraints

- English is always the internal key / source of truth
- URL path params remain in English
- Translation files are loaded and cached in memory (same pattern as existing data)
- Invalid `lang` values fall back to English silently
- GIF URLs and exerciseId are language-independent
- Structure supports adding new languages by creating new `src/data/i18n/{lang}/` folders
